# Contributing to BuildLens

Thanks for your interest in BuildLens. It's a polyglot, event-driven analytics
platform for GitHub Actions, built one reviewable slice at a time. This guide
covers how to set up, what the conventions are, and how to get a change merged.

By participating you agree to keep the project welcoming and harassment-free.
Be respectful; assume good faith.

---

## Ways to contribute

- **Code** - fix a bug, improve a service, or take on a roadmap item (see the
  *How to help* section of the [README](README.md#how-to-help--roadmap)).
- **Docs** - the [README](README.md) is the single source of user-facing docs;
  improvements and clarifications are welcome.
- **Bug reports & feedback** - run the end-to-end walkthrough in the README and
  open an issue if a step doesn't work on your machine.

For anything non-trivial, **open an issue to discuss it first** so we don't
duplicate work or build something that won't be merged.

---

## Getting set up

Follow [Requirements](README.md#requirements) and the
[end-to-end walkthrough](README.md#running-the-full-pipeline-end-to-end) in the
README to get a working local stack. In short:

```bash
make env    # then fill in .env (never commit it - it is gitignored)
make up     # infrastructure + migrations
make dev / make analytics-dev / make ai-dev   # the three services, on the host
```

Read [`AGENTS.md`](AGENTS.md) before changing anything - it lists the invariants
that must not break silently, and [`docs/phases.md`](docs/phases.md) records the
*why* behind each major decision.

## Workflow

1. **Fork and branch.** Name feature branches `feat/<what>`; keep a change
   focused on one thing.
2. **Make your change** with tests.
3. **Run the checks** for the service you touched (below) - your PR should be
   green on that service.
4. **Open a pull request** against `main`, describing what changed and why. Link
   the issue it addresses.

Commit messages: a concise, imperative subject line (e.g. `fix: resume sync after
rate-limit backoff`), with a body explaining the *why* when it isn't obvious.

---

## Quality gates

A pull request should pass the checks for the service it changes:

| Service | Language | Commands |
| ------- | -------- | -------- |
| `gateway/` | Rust | `make fmt` · `make lint` (clippy, `-D warnings`) · `make test` |
| `analytics/` | Java | `make analytics-check` · `make analytics-test` (or `cd analytics && mvn test`) |
| `ai-worker/` | Python | `make ai-check` (ruff format + lint) · `make ai-test` (pytest) |

CI runs these on every pull request.

## The invariants (do not break these silently)

These are enforced by structure, not convention. If a change needs to break one,
call it out explicitly in the PR. Full detail is in [`AGENTS.md`](AGENTS.md).

- **The schema is owned by `infra/migrations`.** No service migrates on its own.
  Create a migration pair with `make migrate-create NAME=...`, edit **both** the
  `.up.sql` and `.down.sql`. Java runs `ddl-auto=validate`; if an entity doesn't
  match the schema, fix the entity - not the database. Python never migrates.
- **Per-service Postgres grants make ownership real.** The gateway writes facts,
  analytics writes derived numbers, the AI worker writes reports, and nobody edits
  the audit log. **If you add a table, add its grant in
  `infra/migrations/000010_grants.up.sql`.**
- **Cross-stream foreign keys are nullable on purpose** (webhooks arrive out of
  order). Within a stream (job → run, step → job) they are strict. Don't "fix"
  the nullable ones.
- **Every re-run attempt is its own `workflow_runs` row**, keyed
  `(github_run_id, run_attempt)`.
- **Events publish through the transactional outbox** and delivery is
  at-least-once - every consumer must be idempotent.
- **Primary keys are `uuidv7()`** generated by the database default; inserts omit
  the id.

## Changing the event contract

The RabbitMQ envelope is a contract shared by three languages (see
[the README](README.md#the-event-contract)). Adding a field is safe - consumers
must ignore unknown fields. Removing or retyping a field is a **breaking** change
that bumps `version`. Update the README's event-contract section and the example
payloads in `contracts/` together.

## Secrets & privacy

- `.env` is gitignored - never commit real secrets. The credentials in
  `docker-compose.yml` / `.env.example` are local-development defaults only.
- The AI worker sends only bounded, **secret-redacted** excerpts to the model,
  never raw log archives or repository source. Keep it that way.
- Found a vulnerability? Do **not** open a public issue - see
  [`SECURITY.md`](SECURITY.md).
